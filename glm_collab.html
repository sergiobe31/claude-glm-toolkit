Give Claude a second model

A cheap, independent model from a different training distribution, wired into Claude Code as a second opinion. It improves your results two ways — better prompts and a real second audit — and nothing it says is trusted until Claude verifies it.

What it does for you

Two levers, plus a bonus pattern. These are problems every developer hits — a second model from a different training distribution is the lever that moves them.

Claude orchestrates and verifies — it has the files, the web and the repo, and it adjudicates every answer. The second model advises and never executes — it reasons over what Claude hands it, from a different training distribution.
Lever 1

① Better prompts

/interceptor captures your project context, finds the gaps in your idea, and — optionally — has the second model red-team the draft. You get a prompt that's complete, constrained and scoped, ready to paste into a coding agent. It returns the prompt; it never runs the task.

skill · /interceptor second-model red-team (optional)
Lever 2

② A second audit

A genuine second opinion that catches what single-model self-review misses. /debate pits your position against the second model under an evidence gate, then synthesizes a verdict stronger than either side — or just ask Claude: “get a second opinion on this code.”

Bonus

Cheap context-grunt: delegate the token-heavy reading

Point the cheap model at the big files; Claude stays in charge and gives the directives. Heavy context under expensive supervision — and, as always, verified before any of it is trusted.

The discipline

Nothing the second model says is trusted until it's verified against ground truth.

Its output is a proposal, not truth. Claude checks it claim-by-claim against file:line / source / trace and tags each one. Verification cuts both ways: it catches a real bug and it clears a false alarm. That's what turns a second model from a liability into an asset.

Real cases — from building this toolkit

Not theory, and not the author's private projects: these happened while building the plugin itself — and the discipline is visible in every one.

1

Choosing the architecture

Lever 2 · /debate

A /debate session weighed packaging this as a native Claude Code plugin against a hand-rolled setup, cross-checked against the official docs. The plugin won: one versioned, installable unit, with the secret handled natively. The decision was stress-tested, not assumed.

decision stress-tested cross-checked vs docs
2

Making it model-agnostic

Live · the discipline on the toolkit itself

The second model recommended keeping DEFAULT_MODEL=auto — the right call. But its stated reason was wrong. Claude verified against PAL's actual source (providers/registries/base.py, providers/openrouter.py) and corrected the mechanism. The conclusion stood; the reasoning didn't. Exactly what "verify, don't trust" looks like.

right conclusion wrong mechanism caught by reading the source

4

The second model was silently capped at 32K

Live · verify, don't trust — on the plumbing

GLM kept refusing files it should have read. Claude traced it to PAL's bundled registry not listing GLM (providers/openrouter.py), so it fell back to a generic ~32K window — a per-call file budget of only ~3.5K tokens. A superset registry restored GLM's real 1M window and the budget jumped to ~268K. Proven by handing GLM the exact batch it had just rejected — and watching it read it.

root cause in the source ~3.5K → ~268K tokens/call verified live
5

Teaching the second model to reason

Live · a verified gap, fixed and proven

GLM-5.2 supports step-by-step reasoning, but PAL's request path never sent OpenRouter's reasoning field — so it never reasoned, and PAL discarded the trace. Claude confirmed the gap in the source (providers/openai_compatible.py), patched it in a minimal, gated fork (only flagged models; byte-identical to upstream otherwise), and proved it live: GLM now returns its <reasoning> trace at xhigh, dialable per call.

gap found in the source minimal gated fork reasoning verified live
6

The second model red-teams its own setup

Live · a verdict stronger than either side

Asked to attack the fork's design, GLM caught a real flaw — a gate that would have changed behavior for 12 models, not one. But its proposed fix would have broken: PAL rejects unknown registry fields (providers/registries/base.py). Claude kept the valid catch, refuted two false alarms with the source and the OpenRouter API, and shipped a design neither side had alone.

real flaw caught broken fix refuted synthesis shipped
